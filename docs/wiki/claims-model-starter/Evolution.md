## 1. Concept origins

The project started as a short brief — what is now `docs/architecture-history/initial_purpose.txt` — describing a six-step pipeline that would walk a business stakeholder from idea to model-ready GitLab project. The brief opens with a role: *"an expert data scientist, business analyst, and consultant focused on a claims organization with a property and casualty insurance company that sells auto and property policies."* This is the domain the intake agent still carries as its persona today, and the six steps in the brief — intake interview, intake report, data collection, data report, initial model website, data science team handoff — map one-to-one onto the pipeline you see in the [Pipeline Overview](Pipeline-Overview).

The brief's worked example is subrogation recovery. A new claims system was capturing less information than its predecessor during the intake and investigation phases, and subrogation recovery rates had dropped. The root causes listed in the brief are concrete: *"inadequate adjuster training on the new system, a poorly designed user interface within the claims system which hinders efficient information capture, and a lack of performance metrics tied to subrogation success for adjusters."* The model solution is a supervised classifier predicting *the probability of successful subrogation* based on claim attributes — precisely the kind of outcome the generated project's initial notebooks explore, and precisely the domain the intake agent probes for when it interviews a stakeholder. A 10% improvement in recovery rates was the target order of magnitude, "likely in the hundreds of thousands or even millions of dollars, depending on the overall claim volume and average claim value."

Three pieces of that brief became architectural constraints that held throughout the project:

1. **"Ask one question at a time."** The brief says it explicitly; the intake agent's LangGraph cycles make it structurally impossible to advance more than one prompt per tick. This is what keeps the interview interview-shaped rather than form-shaped.
2. **"Up to 10 questions, whichever comes first."** The initial `MAX_QUESTIONS=10` cap came directly from the brief. It later rose to 20 (§3 below) when real-LLM runs needed more room to converge.
3. **Governance proportional to value.** The brief sizes the subrogation model at *"hundreds of thousands or even millions of dollars annually"* — a high-stakes recovery model. This is why the pipeline emits governance artifacts *proportional to risk tier and cycle time* (§5 below) rather than a flat checklist; a low-stakes internal tool does not need SR 11-7 paperwork, but a million-dollar-impact classifier does.

**Session 0 (2026-04-10)** adopted the Iterative Session Methodology framework (v2.1) — `SESSION_RUNNER.md`, `SAFEGUARDS.md`, `SESSION_NOTES.md`, the methodology dashboard, and the workstream reference documents under `docs/methodology/`. This is the operating procedure every subsequent session followed. The session also populated `ROADMAP.md` with five milestones (M1 Architecture & Pipeline Design through M5 Orchestrator + Hardening) and `BACKLOG.md` with granular tasks per milestone. `CLAUDE.md` was customized to point at the project's specific task-to-workstream mappings — agent building, prompt engineering, schema definition, pipeline integration.

**Session 1 (2026-04-10)** wrote `docs/architecture-history/architecture-approaches.md` — an exploration of four critical architecture questions (agent coordination, schema evolution, persistence, platform abstraction) with pros/cons for each option. This was pre-canonical-plan work: the choices narrowed later, but the document preserves the alternatives that were considered and rejected. It now lives in the archive as primary-source design archaeology. Reading it today, you can see which options were closed off early (shared-state agent coordination was rejected in favor of explicit handoffs; a single repo-host adapter was rejected in favor of a neutral protocol with per-platform implementations) and which only crystallized later (schema versioning took until Phase 1 to commit to `HandoffEnvelope`; persistence was prototyped in Phase 3B).

## 2. Foundational architecture

**Sessions 2–3 (2026-04-14)** produced `docs/architecture-history/architecture-plan.md` — the canonical 14-phase architecture plan — and the v1 payload schemas. The plan is about 900 lines. Two of its sections became load-bearing invariants:

**§7 — Decoupling invariant.** *The data agent has zero imports from the intake agent. All cross-agent communication goes through the orchestrator adapter.* This is enforced by an AST-walk decoupling test (`tests/data_agent_package/test_decoupling.py`) that parses `packages/data-agent/src/model_project_constructor_data_agent/` and fails the build if any import chain reaches `model_project_constructor.agents.intake`. The invariant exists because the data agent was explicitly designed to be reusable as a standalone query-writing tool (see §4 below), and coupling to intake-specific types would make that reuse impossible. The decoupling test is the structural guarantor — it catches drift that code review would miss.

**§8 — Governance artifact framework.** Artifacts are emitted proportional to `risk_tier` ∈ {low, medium, high} and `cycle_time` ∈ {rapid, standard, extended}, with explicit regulatory mapping (SR 11-7, NAIC AIS). Tier 3 + extended cycle produces the full artifact set — datasheets, model card, monitoring plan, risk register, regulator-facing summary. Tier 1 + rapid produces only the minimal scaffolding. The tier derivation happens inside the intake agent (on every revision cycle, via `GovernanceMetadata`) so that what the website agent produces is a direct function of what the stakeholder described.

Schemas live under `src/model_project_constructor/schemas/`, each tagged with a `schema_version` field. The `HandoffEnvelope` wraps them with versioning metadata so future schema evolution can stay backwards-compatible: an envelope says "I'm carrying an `IntakeReport` at `schemas.v1.intake`," the consumer looks that up in `schemas/registry.py`, and if the version is unknown the pipeline halts with a typed error rather than failing later with a validation mismatch. This matters because the three agents are deployed together today but the design allows them to evolve at different schema cadences tomorrow.

**Phase 5 — Orchestrator (Session 15, 2026-04-15)** made the plan concrete. `src/model_project_constructor/orchestrator/pipeline.py` drives the pipeline sequentially — intake, then data, then website — with each stage's output serialized to a checkpoint before the next stage reads it. `adapters.py` is the sole `IntakeReport ↔ DataRequest` bridge; this is what the §7 decoupling invariant *allows*. `checkpoints.py` persists envelopes and terminal results to `.orchestrator/<run_id>/`. `PipelineStatus` is a `Literal` — `COMPLETE` / `FAILED_AT_INTAKE` / `FAILED_AT_DATA` / `FAILED_AT_WEBSITE` — and partial state is retained on halt so operators can inspect what went wrong. The `*_AT_*` naming is deliberate: a halt identifies *where* the failure happened, which tells an operator which checkpoint to resume from. Automated resume-from-checkpoint is listed in `BACKLOG.md` as a natural follow-on; the scaffolding for it already exists in `CheckpointStore`.

**Phase 6 — Production hardening (Session 16, 2026-04-15)** added observability, configuration, and CI. Two design choices here are load-bearing and worth dwelling on:

1. **Observability via runner composition, not imports.** `make_logged_runner(agent.run, logger=...)` and `make_measured_runner(agent.run, registry=...)` wrap an agent's callable without modifying the pipeline itself. `pipeline.py` has zero imports from the logging or metrics modules — you can grep for it. Observability is opt-in: you compose it at orchestrator construction time, not at pipeline-author time. This keeps the core pipeline trivially testable (no logger mocks needed) and lets different deployments choose different observability layers (the existing `make_logged_runner` uses stdlib `logging`, but a future deployment could swap in OpenTelemetry without touching the core). The same runner-composition shape is what lets the intake agent's fixtures bypass the FastAPI session store: fixtures use a plain `IntakeAgent.run_scripted`, the web UI uses a composed runner with session persistence.

2. **Zero new dependencies.** The hardening plan suggested `structlog` for structured logging and `pydantic-settings` for env-var-driven configuration. Both were rejected in favor of stdlib `logging` with `extra={"context": ...}` for structured fields and a plain `@dataclass` + `os.environ` pattern for config. The rationale (captured in Learning #13): adding a dependency is a one-way door — future sessions must maintain it, CI must install it, version conflicts can surface. `structlog`'s ergonomics are nice but the payload the pipeline emits is already JSON-serializable dicts; stdlib `logging` with an `extra` field plus a JSON formatter covers the use case. The zero-new-dep path was validated by 54 new tests + mypy strict + ruff clean on the first try.

The same session added four-job CI (`lint`, `typecheck`, `test`, `decoupling`), `OPERATIONS.md` as the production runbook, and `TROUBLESHOOTING.md` with one walkthrough per `FAILED_AT_*` path (so an operator who sees `FAILED_AT_DATA` can jump straight to the section that explains what to inspect). Coverage floor raised 93% → 94%.

## 3. Intake agent arc

**Phase 3A — Intake agent core (Session 6, 2026-04-14)** built the intake agent as a LangGraph flow. It interviews a stakeholder one question at a time, rebuilds `IntakeReport` + `GovernanceMetadata` on every revision cycle (so the risk tier reflects the latest answer set, not stale tier from an earlier draft), and stops when either the stakeholder accepts the draft or the budget runs out. Two budgets: `MAX_QUESTIONS=10` (initial; later bumped to 20) and `MAX_REVISIONS=3`. A six-token accept vocabulary in `nodes.py` recognizes stakeholder confirmation across natural phrasings — "yes," "accept," "looks good," "approved," "that's right," "ship it" — so the stakeholder doesn't have to guess the magic word. The agent's fixture-driven CLI lets it run from canned answers for testing, which is what the entire test suite depends on.

The one-question-at-a-time shape is deliberate and traces directly to `initial_purpose.txt`. The brief insisted on it. The LangGraph cycles enforce it structurally: the interview can only advance one prompt per tick, and the cycle boundary is where the revision logic runs. This is what makes the intake experience interview-shaped rather than form-shaped — the stakeholder is not filling out a form, they are having a conversation.

**Phase 3B — Intake agent web UI (Session 7, 2026-04-14)** added a FastAPI + Server-Sent Events + HTMX frontend. SSE was chosen over WebSockets because the interview is unidirectional (agent asks, stakeholder answers, one at a time); SSE needs only a one-line HTML fragment update per turn, which HTMX handles natively without client-side JavaScript. Sessions persist to SQLite so an interview can be paused and resumed. A subtle but important choice: fixtures bypass the database entirely — they test the agent's LangGraph flow without touching disk state — which preserved the testing patterns established in Phase 3A while adding a production-shape web surface. Fixture statelessness means the test suite runs in under 10 seconds; session-state tests are a separate, smaller suite that exercises the SQLite persistence path explicitly.

**Scope B-2 — Real-Anthropic intake (Session 26, 2026-04-16)** extended the pipeline script with a third mode: `--llm both`. Previously the intake stage was always fixture-driven — the pipeline script ran the intake agent's `run` method against a pre-built `IntakeReport` fixture. Session 26 introduced `IntakeAgent.run_scripted(...)`, which calls real Anthropic for *questions* but consumes fixture-supplied *answers*. This lets you exercise the real LLM's question-generation path without needing a human in the loop. The `--intake-fixture PATH` flag is required when `--llm=both`.

A subtle piece of this work is the `_draft_incomplete_from_exception` adapter (inline in `scripts/run_pipeline.py`). It converts exhausted-script `RuntimeError` (the fixture ran out of answers before Claude converged), rate-limit errors, and pydantic validation failures into a typed `IntakeReport(status="DRAFT_INCOMPLETE")` so the orchestrator halts cleanly with `FAILED_AT_INTAKE` instead of crashing with an untyped exception. The reason code is the exception class name; the full message goes into `missing_fields[0]`. This pattern — *convert lower-level crashes into typed halt states at the boundary* — matches the orchestrator's overall shape, where `PipelineStatus` is always one of four known values and there is no "unknown error" state.

**Cap bump — `MAX_QUESTIONS` 10 → 20 (Session 27, 2026-04-17)** is the intake-agent arc's first substantive tuning, and the one whose story is worth telling in full because it illustrates a recurring pattern. Session 26's first live B-2 run against a 10-qa-pair subrogation fixture produced a rich Claude-drafted report but finished with `DRAFT_INCOMPLETE`. The status-decision code (`nodes.py:142`) is:

```python
status = "COMPLETE" if accepted and not missing else "DRAFT_INCOMPLETE"
```

And the auto-append at `nodes.py:129-134` fires `questions_cap_reached` whenever `questions_asked >= MAX_QUESTIONS` and `believe_enough_info` is still false. So the happy path to `COMPLETE` requires the LLM to flip `believe_enough_info=true` *before* hitting the cap. Against the 10-qa-pair fixture, Claude Opus 4.7 wanted more detail than 10 questions could supply — specifically, it flagged three gaps it could not resolve: latency SLA (is this real-time at FNOL or after intake completion?), average recovery per successful subrogation (how many recoveries per year is $30M/yr?), and fairness/bias plan (are protected attributes being used?). Claude kept asking; the fixture ran out; the cap fired; the status went to `DRAFT_INCOMPLETE`.

The fix was two-part: raise the cap to 20 *and* extend the fixture to 15 qa_pairs pre-answering the three gaps. The live `run_b2_complete` after the fix reached `Status: COMPLETE` with `questions_asked=10` — Claude converged inside the old cap *once the fixture answered the right questions*. The wider cap is not for padding; it is for fixtures (and real stakeholders) that don't pre-answer as thoroughly.

This bump produced Learning #21 — *before writing a plan criterion like "live LLM + fixture → COMPLETE status," read the status-decision code.* The plan's criterion had assumed "COMPLETE = happy path" without reading `nodes.py:142`; the reality was that the invariant required the LLM to flip `believe_enough_info=true` before the cap. Spending two minutes on that read up front would have surfaced the constraint before a $0.30 live run.

The cap bump also surfaced something structural: a production constant (`MAX_QUESTIONS`) had leaked into test fixtures (via `MAX_QUESTIONS + 1` sizing arithmetic — `intake_question_cap.yaml` needs one more answer than the cap to demonstrate the cap firing), dict-literal exports (`CAPS = {"max_questions": MAX_QUESTIONS}` exposed for a test to assert against), and prose in several wiki and ops surfaces. A plain `grep MAX_QUESTIONS` missed the indirect coupling paths. Learning #22 emerged — *grep for BOTH the UPPER_CASE constant name AND the lower_case field form* when bumping a production constant. The drift across the prose surfaces took until Sessions 40–42 to fully close (see §7 below).

## 4. Data agent arc

**Phase 2A — Data agent core (Session 4, 2026-04-14)** built `DataAgent` as a LangGraph flow. It takes a `DataRequest`, generates SQL queries (via the LLM, with the prompt constrained by the request's schema), runs quality checks, and produces a `DataReport`. `sqlparse`-based validation catches syntax errors before queries reach a database — the agent emits SQL that compiles, even if the semantics need review. The AST-walk decoupling test is the architectural guardrail; `DataAgent.run(DataRequest) -> DataReport` is the minimal entry point.

**Phase 2B — Standalone package (Session 5, 2026-04-14)** refactored the data agent into `packages/data-agent/` with its own `pyproject.toml`. The motivation is in `initial_purpose.txt` verbatim — the brief noted that the data agent *"would likely be useful for just writing queries in general"* for analyst teams. *"Many analysts spend significant time writing queries; speeding this up would enable exploratory analysis that is currently infeasible."*

A standalone package makes reuse possible in a way that a subpackage of the orchestrator does not: an analyst team can depend on `model_project_constructor_data_agent` without pulling in the orchestrator, the website agent, FastAPI, or LangGraph's intake-specific nodes. The package added `AnthropicLLMClient`, a Typer CLI (`data-agent` as an entrypoint), and Python API docs. Coverage floor raised 80% → 90%. The "reusable as analyst tool" design principle is also why the data agent carries `--db-url` as a first-class CLI argument: an analyst's use case is *"point at my warehouse, ask for a query"* and requiring them to round-trip through an `IntakeReport` would defeat the purpose.

**Scope B-1 — Real Anthropic wiring (Session 24, 2026-04-16)** connected the real LLM path to the pipeline script. `scripts/run_pipeline.py` gained `--llm {none,data}` (default `none` preserves Scope A behavior), `--model ID` (default `claude-opus-4-7`), and `--db-url URL`. `build_data_runner` returns either a fixture-serving closure (`--llm none`) or `DataAgent(AnthropicLLMClient(model=...), db=...).run` (`--llm data`). The intake stage stayed fixture-driven at this phase; Scope B-2 wired the real intake path.

The model-choice decision on this session is worth noting because it set a precedent. The scope-b plan's §8.2 recommended `claude-sonnet-4-6` for the first live run to save cost. The user pushed back: if the output quality was poor, *"was it the model?"* would become a confounding variable. Pilot cost at ~$0.50/run on opus vs ~$0.10/run on sonnet is trivial compared to the cost of a muddled conclusion. The first-impression run used `claude-opus-4-7`. This became Learning #20 — *for the first quality-judging run, eliminate the model as a variable. Iterate to cheaper models once the output shape is validated.*

## 5. Website agent arc

**Phase 4A — Website agent core (Session 8, 2026-04-14)** built the website agent as a LangGraph flow with three nodes: `CREATE_PROJECT` (creates the repo), `SCAFFOLD_BASE` (writes the baseline file tree — `.qmd` analysis files, `src/` module stubs, README, CI config), `INITIAL_COMMITS` (groups the scaffold into meaningful commits — "scaffold", "governance artifacts", "CI setup" — so the git history is readable). The output is a draft repository with placeholders clearly marked where human judgment is required.

The brief called for a *"draft, not a finished product."* The scaffolding bias is toward making the data science team's first-day experience productive. Three small choices reflect this:

- Placeholders are `# TODO(data-science-team): ...` comments with specific prompts, not generic `FIXME`s. A notebook cell says `# TODO(data-science-team): Decide whether to use SMOTE, class weights, or stratified sampling. See model_card.md §fairness.`, not `# FIXME implement this`. The comment tells the reader what decision to make and where to look for context.
- `.qmd` files (Quarto, not Jupyter notebooks) are chosen so the output renders as a website without a local Python kernel. A reviewer can read the draft analysis in a browser; only an implementer needs the runtime.
- The `src/` module stubs have type annotations and docstrings but raise `NotImplementedError` in their bodies. This gives mypy something to check and gives the implementer a clear signal that the stub has not been filled in yet.

**Phase 4B — Website agent governance (Session 9, 2026-04-14)** is the website-agent arc's most architecturally-interesting session. It added governance artifact scaffolding proportional to `risk_tier` and `cycle_time` per architecture-plan §8 — the framework described in §2 above. Tier 3 + extended produces about 10 artifacts (datasheets, model card, monitoring plan, risk register, regulatory-mapping summary, sign-off template, etc.); tier 1 + rapid produces 2 or 3 (lightweight model card + basic datasheet). The regulatory mapping is explicit: `SR_11_7` (federal supervisory guidance on model risk management) and `NAIC_AIS` (insurance-specific model governance) both get pre-formatted sections in the tier-3 outputs.

Two design decisions from this session shaped the rest of the codebase:

1. **`is_governance_artifact(path)` as the single source of truth.** A naive implementation would have stored the governance manifest in agent state, accumulating it as each node ran (each node would add its own artifacts to `state["governance_manifest"]`). Instead, the classifier function determines whether any given path *is* a governance artifact; the `GovernanceManifest` is *derived* by applying the classifier to the final file tree. This is the difference between storing a fact and recomputing it from ground truth. State bookkeeping drifts when new nodes are added — someone forgets to append to the manifest — but a classifier cannot drift because it is called on the file tree itself. This became Learning #3 — *derive manifest/audit data by classifier function, not by storing it in state during each node.*

2. **Helper functions in a sibling module, not an expansion of the existing one.** `governance_templates.py` sits next to `templates.py`. The review-time diff stays local to the new governance concerns; the "before vs after" phase-split is visible in the file tree. Learning #2 codified this pattern for future LangGraph phase extensions: *when extending a LangGraph pipeline with new nodes, put the helper functions in a sibling module rather than expanding the existing module.*

Session 9 also added the first repository adapter — `PythonGitLabAdapter` — implementing what was then called `GitLabClient`. Retry and exponential backoff wrap repo operations so transient network failures don't abort a live run. The retry policy has visible seams (kwargs on the adapter constructor) so a test can pass a no-op `sleep` without monkeypatching the production backoff timer (Learning #4: *plumb test-needed kwargs through the production factory, but only if the kwarg is also valuable in production; otherwise accept a construction hack in tests rather than over-engineer production*).

## 6. Platform abstraction arc

The website agent initially assumed GitLab. Sessions 10–14 generalized it to a repo-host-agnostic design, with GitHub as the second supported platform. This was done as a four-phase plan (`docs/architecture-history/github-gitlab-abstraction-plan.md`) rather than a single refactor, because a rename mixed with a behavior change would have made the review diff impossible to read.

**Phase A — Neutral rename (Session 11).** `GitLabClient → RepoClient`, `GitLabTarget → RepoTarget`, `GitLabProjectResult → RepoProjectResult`. The `project_id` field widened from `int` (GitLab's numeric ID) to `str` (so GitHub's `owner/repo` form fits). This phase touched 26 files and was deliberately the only rename phase — the commit contains nothing but renames, no behavior changes. Learning #8 emerged here: *a grep-based inventory for a rename plan must include the type, field, state key, and docstring surfaces separately.* Each has a different grep pattern; conflating them gives misleading counts.

**Phase B — GitHub Actions CI template (Session 12).** `render_github_actions_ci` as a sibling to the existing GitLab CI renderer (`render_gitlab_ci`). A new `ci_platform` field flows through website agent state and the governance manifest, so the emitted CI manifest can differ from the repo host (primarily for fake-mode cross-platform testing — if you're testing on GitLab fake-mode but want to preview what the GitHub CI file would look like, `--ci-platform github` decouples the two).

**Phase C — PyGithub adapter (Session 13).** `PyGithubAdapter` implementing the neutral `RepoClient` protocol. `PyGithub` became a dependency — LGPL-3.0 licensed, but usage is pure API (no redistribution of modified `PyGithub` code), so LGPL compliance is automatic via Python's import mechanism. A gotcha from Session 12 had claimed `PyGithubAdapter` should not inherit from `RepoClient` — but the existing `gitlab_adapter.py:41` already inherited, and Learning #12 emerged from this: *when a gotcha contradicts the codebase pattern, mirror the codebase and document the resolution.* A gotcha that names a past fact is a claim about the state at that moment; if the codebase has evolved since, the codebase wins.

**Phase D — Host selection CLI (Session 14).** `--host gitlab|github` on the website CLI. Dual-adapter selection at invocation time. Coverage floor raised 90% → 93%.

**Session 22 — First live end-to-end smoke (2026-04-16)** ran the full pipeline against a real GitLab project for the first time. The smoke test surfaced a bug that none of the Phase 4B/5 tests had caught: `scripts/run_pipeline.py:119` passed `url=` to `PythonGitLabAdapter(...)`, but the constructor's keyword-only parameter is `host_url=`. `--live --host gitlab` had never successfully run — it would have raised `TypeError: __init__() got an unexpected keyword argument 'url'` on the first invocation. A two-character fix (`url=host_url` → `host_url=host_url`) unblocked the path, and a real project was created at `https://gitlab.com/rmsharp-modelpilot/subrogation-pilot` with 38 files, 10 tier-3-moderate governance artifacts, and regulatory mapping on `SR_11_7` + `NAIC_AIS`. The session filed five additional findings (later addressed in Sessions 25, 28, 29, 30, 31, 32) covering CI scope gaps, namespace validation, GitHub host-url wiring, and operations-doc drift. *This is the session that revealed how much latent bit-rot accumulates in paths that are structurally tested but never exercised end-to-end.*

**Session 28 — `validate_namespace` fail-fast (2026-04-17)** closed one of those findings. `MPC_NAMESPACE` takes a group path (`my-org/subgroup`), not a URL — but a URL-form value had surfaced in Session 22 as a generic `"group lookup failed: 404"` from the GitLab adapter, leaving operators to guess which config was wrong. Session 28 added a module-level `validate_namespace` helper that raises `ConfigError` on `http://` or `https://` prefixes (case-insensitive, whitespace-trimmed) before any agent runs. The failure now surfaces at `[2/5] Building pipeline config...` with a message naming the received value and the expected path form. 13 new tests pinned the validator behavior. This fail-fast pattern — *validate at config boundary, halt with a typed message, not a generic 404* — matches the orchestrator's `FAILED_AT_*` halt-state shape.

**Session 30 — GitHub branch parallel-fix completeness (2026-04-17)** addressed a paired bug in the GitHub branch of `scripts/run_pipeline.py:273-278`. The branch passed `token=token` to `PyGithubAdapter`, but the adapter's kwarg is `private_token=`; it also never read `MPC_HOST_URL`, so GitHub Enterprise users would silently hit public `api.github.com`. The two bugs together meant `--live --host github` had never worked. The session was framed in its handoff as "parallel to Session 22's GitLab fix," but the parallel ran deeper than the named symbol — there were two defects, not one. Learning #23 emerged: *when a handoff describes a fix as parallel to Session X, read both sibling call sites side-by-side before editing.* A 10-second grep of both adapters' constructor signatures + both call sites catches the whole class in one pass.

**Session 32 — `cli.py` GitLab default URL (2026-04-17)** changed `GITLAB_DEFAULT_HOST_URL` at `src/model_project_constructor/agents/website/cli.py:39` from `"https://gitlab.example.com"` (an RFC-2606 placeholder that does not resolve) to `"https://gitlab.com"` (the public GitLab API URL). This brought the CLI default in line with the four other surfaces declaring the same default (`orchestrator/config.py:33`, `scripts/run_pipeline.py:105,283`, `.env.example:18-19`, `OPERATIONS.md:24`). The test restructure around this fix produced Learning #25 — *when two platform-symmetric tests are in the same file and one asserts defaults while the other asserts overrides, the asymmetry is usually a bug in test design.* The GitHub sibling test asserted defaults (no `--host-url` passed); the GitLab test asserted overrides (explicit `--host-url` passed); neither pinned the default for GitLab. Fix: restructure GitLab to default-path, add a separate explicit-override test. Session 37 later added the symmetric GitHub explicit-override test to close the loop.

## 7. Methodology arc

The methodology framework itself evolved across the project. Every session is governed by `SESSION_RUNNER.md` and `SAFEGUARDS.md`; failures teach the framework something the next session inherits. The Learnings table in `SESSION_RUNNER.md` (32 learnings at the time of writing) is where the patterns accumulate.

**Session 17 — Pilot readiness (2026-04-15)** worked through every one of `architecture-plan.md` §14's 46 acceptance criteria and declared the pilot gate met. Along the way it fixed 62 pre-existing ruff errors (56 auto-fixable + 6 manual) and three CI failures (mypy missing `--extra ui`, decoupling job below the coverage floor, CLI tests failing on ANSI color codes — resolved via `click.unstyle`). The `[0.1.0 — Pilot Ready]` tag in `CHANGELOG.md` dates from this session: 422 tests at 97.18% coverage, both platform adapters passing structural + integration tests, CI green across all four jobs.

**Session 18 — Tutorial (2026-04-16)** wrote `scripts/run_pipeline.py` (265 lines) and `docs/tutorial.md` (six steps, restructured during user testing to start with fixture creation — the first draft started with the programmatic API, and user testing made it clear that a fixture was the natural first object to understand). The session also made a small but durable content correction: *likelihood* replaced with *probability* in five files wherever the context was `P(event)`. Likelihood has a specific statistical meaning (the function `L(θ|x)` viewed as a function of parameters given data); using it for `P(event)` is a common conflation in LLM-generated text. This became a project-local preference that subsequent sessions carried forward — the intake agent's persona specifically uses `probability` when referring to `P(event)`, and `BACKLOG.md` carries a pending "statistical terminology glossary" item to make this preference enforceable via the agent system prompts.

**Sessions 19, 20A, 20B — Wiki expansion (2026-04-16)** wrote the 21 wiki pages you see in the sidebar today — Home, Getting Started, Pipeline Overview, Governance Framework, Agent Reference, Schema Reference, Worked Examples, and the rest. Paste it. Re-verify only what the predecessor could not have verified in advance (the `--help` output is still current, the section boundary hasn't moved).

The BACKLOG item for `--ci-platform` in `docs/tutorial.md` had pointed at §5 (which is where the tutorial introduces live-mode). Learning #28's second validation: *respect handoff intent, audit handoff specifics against the actual code.*

**Session 37 — Stub-fumble safeguard first firing (2026-04-17)** is a small session with an outsized methodology implication. 4. **Planning-doc archive.** Completed plans move to `docs/architecture-history/` with an in-file banner; active plans stay in `docs/planning/`.
5. **`wiki/Changelog.md` content unchanged** — only an opener was added; the page's release-shaped summary still fits its user-facing audience.
