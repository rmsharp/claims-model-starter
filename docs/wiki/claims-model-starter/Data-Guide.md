# Data Guide

## How data flows through the pipeline

1. **Intake Agent** captures what data the model needs: target variable, candidate features, population, time range
2. **Data Agent** generates SQL queries to collect the data, plus quality-check queries to validate expectations
3. **Website Agent** packages the queries and documentation into the generated repository

The generated repository contains three data-related components: queries, datasheets, and the data loading module.

## Queries

### Primary queries (`queries/primary/`)

Each `.sql` file is a standalone query generated by the Data Agent to collect training data. The query names and purposes are documented in:

- `data/README.md` -- data dictionary listing each query with expected row counts and purpose
- `reports/data_report.md` -- human-readable summary of the Data Agent's work
- `reports/data_report.json` -- machine-readable DataReport

### Quality-check queries (`queries/quality/`)

Organized by primary query name, each subdirectory contains validation queries. These check data expectations identified during intake, such as:

- Expected row count ranges
- Null rates for key columns
- Value distributions for categorical fields
- Date range coverage
- Join completeness

Run quality checks before using primary query results to ensure data meets expectations.

## Datasheets (`data/datasheet_<query>.md`)

Each primary query gets a [Gebru et al. 2021](https://arxiv.org/abs/1803.09010) datasheet covering:

| Section | What it documents |
|---------|------------------|
| Motivation | Why this dataset was collected |
| Composition | What instances represent, relationships between fields |
| Collection process | How the data was generated (query logic) |
| Preprocessing | Any transformations applied |
| Uses | Intended use cases and limitations |
| Known biases | Potential sources of bias in the data |
| Maintenance | Who maintains the data source, update frequency |

These datasheets are scaffolds -- the data science team should update them with actual findings after running the queries.

## Data loading module (`src/<project>/data_loading.py`)

The generated module contains:

- `PRIMARY_QUERY_NAMES` -- list of query names from the DataReport
- `read_sql(query_name)` -- stub to read a `.sql` file and execute it
- `load_primary()` -- stub to load all primary query results

The team implements these functions to connect to their data warehouse:

```python
from <project>.data_loading import load_primary

# After implementing the stubs:
df = load_primary()
```

## Data expectations

The DataReport includes two lists that help the team validate their data:

- **Confirmed expectations** -- Data characteristics the Data Agent verified (e.g., "date range covers 2020-2024")
- **Unconfirmed expectations** -- Characteristics the agent could not verify and the team should check manually

These are preserved in `reports/data_report.md` and `reports/data_report.json`.

## Data quality concerns

The DataReport may flag data quality issues discovered during query generation:

- Missing tables or columns
- Unexpected join cardinality
- Date gaps or coverage issues
- Known data quality problems in source systems

Review `reports/data_report.md` > "Data Quality Concerns" before beginning analysis.

## The Data Agent as a standalone tool

The Data Agent is designed to be reusable beyond the pipeline. Analyst teams can use it independently:

```bash
# CLI
model-data-agent run --request request.json --output report.json

# Python API
from model_project_constructor_data_agent import DataAgent
report = DataAgent().run(data_request)
```

The Data Agent accepts a `DataRequest` (target, granularity, features, population, time range) and produces a `DataReport` without any dependency on the Intake Agent or its schemas. This decoupling is enforced by an AST-based CI test.

## Providing a data source inventory

`DataRequest` accepts an optional `data_source_inventory: DataSourceInventory | None` field. When populated, the Data Agent renders a summarized block into the query-generation prompt, nudging the LLM to prefer inventory-named tables over inventing ones. Each generated `PrimaryQuery` records the inventory entries it referenced under `inventory_entries_used`, giving auditors a provenance trail.

Three producer classes can populate an inventory today:

- **Curated** — a team-maintained JSON file following `DataSourceInventory`. See `tests/fixtures/sample_curated_inventory.json`.
- **Automated** — probe a live database's `information_schema` via `probe_information_schema` or the `model-data-agent discover` CLI.
- **Interview** — convert stakeholder-named systems from an `IntakeReport`'s `qa_pairs` (reference implementation deferred until pilot demand).

```python
import json
from pathlib import Path
from model_project_constructor_data_agent import DataSourceInventory

inventory = DataSourceInventory.model_validate(
    json.loads(Path("curated_inventory.json").read_text())
)
request = DataRequest(..., data_source_inventory=inventory)
```

When both `database_hint` and `data_source_inventory` are set, the inventory wins (richer signal); the hint is still passed as context. An empty inventory (`entries=[]`) is treated identically to `None`. Large inventories are truncated in the prompt to the top 20 entries by `relevance_score`, with a trailing note naming the dropped count.
