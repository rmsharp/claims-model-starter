# Development Workflow

This page describes how a data science team works in a generated claims-model-starter repository.

## What you receive

When the pipeline completes, your team gets a repository containing:

- **Quarto narratives** (`analysis/`) pre-seeded with business context, data documentation, and analysis scaffolds
- **Python modules** (`src/`) with function stubs for data loading, features, models, and evaluation
- **SQL queries** (`queries/`) for data collection and quality checks
- **Tests** (`tests/`) with scaffolded test stubs
- **Governance artifacts** (`governance/`) proportional to model risk
- **CI pipeline** that runs lint, tests, and governance schema validation

## Getting started

### 1. Clone and install

```bash
git clone <project-url>
cd <project-name>
uv sync
```

The generated `pyproject.toml` includes `pandas`, `scikit-learn`, and `sqlalchemy` as core dependencies, plus `pytest`, `pytest-cov`, and `ruff` as dev dependencies.

### 2. Review the intake and data reports

Start with the human-readable reports:

```bash
# What the stakeholder asked for
cat reports/intake_report.md

# What data is available
cat reports/data_report.md
```

These explain the business problem, model approach, and data queries generated by the pipeline.

### 3. Run the existing queries

The primary queries in `queries/primary/` are ready to run against your data warehouse. Quality-check queries in `queries/quality/` validate data expectations.

```python
# src/<project>/data_loading.py already lists the query names
from <project>.data_loading import PRIMARY_QUERY_NAMES
```

### 4. Implement the function stubs

Each module in `src/` contains stubs that raise `NotImplementedError`. Implement them in this order:

| Order | Module | Key function | What to implement |
|-------|--------|-------------|-------------------|
| 1 | `data_loading.py` | `load_primary()` | Database connection, query execution |
| 2 | `features.py` | `build_feature_matrix()` | Feature engineering from raw data |
| 3 | `models.py` | `train()`, `predict()` | Model training and inference |
| 4 | `evaluation.py` | `evaluate()` | Metric computation |
| 5 | `fairness/audit.py` | `subgroup_metrics()` | Fairness analysis (if applicable) |

### 5. Fill in the Quarto narratives

The `.qmd` files in `analysis/` are pre-seeded with sections from the intake report. As you implement each module, fill in the corresponding narrative:

| Narrative | Depends on |
|-----------|-----------|
| `01_business_understanding.qmd` | Review only -- content is from intake |
| `02_data.qmd` | `data_loading.py` implementation |
| `03_eda.qmd` | Data loaded and explored |
| `04_feature_engineering.qmd` | `features.py` implementation |
| `05_initial_models.qmd` | `models.py` implementation |
| `06_implementation_plan.qmd` | `evaluation.py` results |
| `99_extensions.qmd` | Ideas captured from all prior sections |

Render with [Quarto](https://quarto.org/):

```bash
quarto render analysis/
```

### 6. Run tests

The test stubs validate that imports work and stubs raise `NotImplementedError`. As you implement functions, update the tests:

```bash
uv run pytest
```

The CI pipeline runs `ruff check`, `pytest`, and governance schema validation on every push.

## Code quality standards

The generated project includes:

- **ruff** -- Linting and formatting. Rules: E (pycodestyle errors), F (pyflakes), I (isort), UP (pyupgrade), B (bugbear), SIM (simplification).
- **pytest** -- Testing framework with coverage reporting.
- **Pre-commit hooks** -- ruff lint + format, model registry schema validation.

Install the hooks:

```bash
uv run pre-commit install
```

## Governance workflow

As the model evolves, maintain the governance artifacts:

1. Update `governance/model_card.md` with actual model details, evaluation results, and ethical considerations
2. Record every significant change in `governance/change_log.md`
3. Update `governance/model_registry.json` if the model type, risk tier, or target variable changes
4. Complete the `data/datasheet_<query>.md` files with actual data characteristics
5. For tier 3+ models: fill in the three-pillar validation plan and monitoring metrics
6. For tier 2+ models: complete the impact assessment with actual risk analysis

## Branch strategy

The generated repository arrives with a single commit on the default branch. The team should adopt their standard branching strategy (e.g., feature branches with PRs). The CI pipeline runs on push to the default branch and on pull requests.
