### 2026-04-19 — Data-agent JSON parser tolerates prose around markdown fences (Session 53)

Fixes BACKLOG item 1 (the second finding from Session 51's live-LLM round-trip, `run_id=run_b1_resume_live_1776570556`). `claude-sonnet-4-6` returned the quality-check list wrapped in ` ```json … ``` ` markdown fences, and the data-agent's `_extract_json` helper raised `LLMParseError` because its fence-strip regex required the entire stripped response to be a single fenced block (`^…$` anchors). In practice, models sometimes add prose before or after the fence (an introductory sentence, a trailing explanation) — the anchored regex did not match, the fence passed through to `json.loads`, and the data agent's graph crashed with `EXECUTION_FAILED`. Session 53's fix restructures `_extract_json` into a two-phase parse: try bare `json.loads` first (fast path for already-clean JSON) and, on `JSONDecodeError`, search for a fenced block anywhere in the response and retry with its contents. Fence-search uses `re.search` without anchors so prose before and/or after the fence no longer blocks parsing. The bare-parse error is preserved as the cause of any final `LLMParseError` so debug output names the real decode failure, not a synthetic one.

- **Changed:** `packages/data-agent/src/model_project_constructor_data_agent/anthropic_client.py` — `_extract_json` restructured to bare-parse-first + fence-fallback. `_CODE_FENCE` regex loosened from `^\`\`\`(?:json)?\s*\n(.*?)\n\`\`\`\s*$` (anchored, match-only-entire-response) to `\`\`\`(?:json)?\s*\n?(.*?)\n?\`\`\`` (unanchored, searchable; optional newlines on both sides of the fenced content). Module docstring updated to describe the new parse order and cite the Session 51 `run_id` as the tracked bug. Docstring of `_extract_json` itself names the invariant: bare-parse-first, fence-search-fallback, raise only if both fail.
- **Added:** 5 regression tests in `tests/data_agent_package/test_anthropic_client.py` — `test_extract_json_prose_before_fence` (the original bug shape), `test_extract_json_prose_after_fence` (trailing explanation), `test_extract_json_prose_before_and_after_fence` (both sides), `test_extract_json_fence_without_language_tag_and_prose` (``` ``` without `json` tag + leading prose), `test_extract_json_bare_json_still_parses` (fast-path regression: bare JSON with leading/trailing whitespace + bare arrays). ### 2026-04-19 — Status-aware resume demotion: FAILED envelopes re-execute their stage (Session 52)

Fixes BACKLOG item 1 (filed by Session 51's live-LLM round-trip, `run_id=run_b1_resume_live_1776570556`). `determine_resume_point` previously used only `CheckpointStore.has()` existence checks, so a `DataReport.json` on disk with `status=EXECUTION_FAILED` — the halt artifact of a `FAILED_AT_DATA` run — was treated as a completed handoff and the resume point returned `"website"`, handing the FAILED report to the website agent. The symmetric bug existed for `IntakeReport` with `status=DRAFT_INCOMPLETE`. Session 52's fix: `determine_resume_point` loads saved `IntakeReport` / `DataReport` payloads and demotes the resume point to `"intake"` / `"data"` when the saved status is not `"COMPLETE"`. `ResumeInconsistent` contract stays narrow (missing-predecessor cases only) — a FAILED envelope is a legitimate halt artifact, not structural corruption. `RepoProjectResult` is NOT symmetric: its FAILED case is handled by `_handle_already_complete` with opt-in-to-retry UX because the website stage has irreversible side effects (GitLab/GitHub project creation).

Added `_is_saved_payload_complete(store, run_id, payload_type)` private helper. Docstring updated to cross-reference plan §11 risk #8 and explain the asymmetry with `RepoProjectResult`.
