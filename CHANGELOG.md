### 2026-04-20 — Wiki publish mechanism — prevent repo-to-wiki drift (Session 64)

Closes BACKLOG item 3 (the follow-up Session 63 split off after its one-shot manual sync). Ships the drift-prevention infrastructure Session 63 identified as missing: a scripted publish step + a session close-out hook that fires when any wiki source file changes. Scope decision (operator-approved): **option (b) from Session 63's three-option menu — script + close-out hook**, not script-only (which failed in 40 sessions) and not CI-on-push (overkill for infrequent changes, would need a PAT secret to push to the separate wiki repo). The script `scripts/publish_wiki.sh` follows the house style established by `scripts/render_tutorial.sh` (bash strict mode, env-var configuration, dependency check, documented exit codes, instruction-rich error messages). Design: idempotent (rsync from `docs/wiki/claims-model-starter/` → `$WIKI_CLONE` (default `~/Development/claims-model-starter.wiki`) → `git add -A` → commit-if-changed → `git push origin master`), so invoking it unconditionally is safe — when source and clone are in parity it exits cleanly with "no changes to publish" in seconds. Multi-layer sanity: verifies the `WIKI_CLONE` target is an actual clone, that its `origin` URL contains `claims-model-starter.wiki` (protects against mis-configured WIKI_CLONE pointing at the wrong repo), that the branch is `master` (GitHub wiki repos only support this legacy branch name), and that the clone has no uncommitted changes via `git status --porcelain` (the authoritative, self-refreshing dirty-check — Session 64 caught that `diff-index --quiet HEAD --` can false-positive on stat-cache staleness). Wiki-clone-not-present is a hard failure with a printed `git clone` command; no auto-clone (one-way-door action on a second repo). Ambient git auth (SSH / credential helper / gh CLI) handles the push; no PAT hard-coded. Commit message on the wiki side cites the repo's source commit SHA for provenance (`docs: sync wiki from model_project_constructor@<sha>`). SESSION_RUNNER.md Phase 3E gains a conditional `Wiki sync check` sub-step naming the guard (`git diff --name-only <previous-HEAD>..HEAD -- docs/wiki/claims-model-starter/`) and pointing at the script; the script's idempotency means a session that touches no wiki files can run it without side effects, so the guard is more of a "when to care" signal than a strict precondition.
